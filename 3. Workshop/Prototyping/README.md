# Prototyping

Quick experiments and proof-of-concept work — the Workshop's active build space. This area holds active prototyping efforts: quick experiments, proofs of concept, technical spikes, and anything being built right now. Prototyping is lean and fast; documentation is sufficient for continuity, not for posterity.

## What belongs here

- Proof-of-concept logs and notes
- Technical spike results
- Build instructions and environment setup records
- Experiment outcomes and observations
- Active project notebooks for projects still in exploratory phase

## What does NOT belong here

- Project ideas not yet started → [[../Project Ideas/README.md|Project Ideas]]
- Completed projects → [[../../7. Archive/Completed Projects/README.md|Completed Projects]]
- Research notes → [[../../1. Study/Research/README.md|Research]]
- Stable reference material → [[../../2. Library/General Reference/README.md|General Reference]]

## Tips

- **Log experiments.** Every prototype session should have a brief log: what was tried, what happened, what's next. Even failed experiments are valuable.
- **Mark status.** Use `status: active`, `status: paused`, or `status: completed` properties. Active prototypes should not accumulate unmarked paused work.
- **Disposable by design.** Prototypes are meant to be transient. When a prototype graduates to a real project, extract the learnings and archive or discard the prototype artifacts.
- **Keep it lean.** Do not over-document prototypes. Enough detail to reproduce and enough context to resume — no more.
